# 📋 GiftPick 요구사항 명세서 (SRS)

**문서 버전**: v1.0  
**작성일**: 2026-06-29  
**프로젝트명**: GiftPick – 생일 선물 선택 플랫폼

---

## 1. 목적 및 범위

### 1.1 목적
본 문서는 GiftPick 서비스의 기능적·비기능적 요구사항을 정의하여, 개발팀과 이해관계자 간 명확한 커뮤니케이션 기준을 제공합니다.

### 1.2 범위
- **포함**: 웹 기반 선물 이벤트 생성·관리, 선물 선택 기능, 사용자 인증/인가, REST API
- **제외**: 모바일 네이티브 앱, 실제 결제 기능, 배송 서비스 연동

### 1.3 용어 정의

| 용어 | 설명 |
|------|------|
| Giver | 선물 이벤트를 생성하는 사람 (관리자 역할) |
| Recipient | 선물을 선택하는 수혜자 |
| GiftEvent | Giver가 생성하는 선물 이벤트 |
| GiftOption | 이벤트 내 선택 가능한 선물 항목 |
| GiftSelection | Recipient의 최종 선물 선택 기록 |
| JWT | JSON Web Token, 인증에 사용되는 토큰 |

## 2. 기능적 요구사항 (Functional Requirements)

### 2.1 사용자 인증 및 계정 관리

#### FR-AUTH-001: 회원가입
- **설명**: 사용자는 이메일, 비밀번호, 사용자명, 역할(GIVER/RECIPIENT)을 입력하여 계정을 생성할 수 있어야 한다
- **입력**: `username`, `email`, `password`, `role`
- **출력**: 회원가입 성공 메시지 및 자동 로그인 처리
- **조건**: 이메일 중복 불가, 비밀번호 최소 8자 이상
- **우선순위**: P0

#### FR-AUTH-002: 로그인
- **설명**: 등록된 사용자는 이메일과 비밀번호로 로그인하여 JWT 토큰을 발급받을 수 있어야 한다
- **입력**: `email`, `password`
- **출력**: JWT Access Token, 사용자 정보(id, email, role)
- **조건**: 잘못된 자격증명 시 오류 메시지 표시
- **우선순위**: P0

#### FR-AUTH-003: 로그인 사용자 정보 조회
- **설명**: 로그인된 사용자는 자신의 계정 정보를 조회할 수 있어야 한다
- **입력**: Authorization Header (Bearer JWT)
- **출력**: 사용자 id, username, email, role
- **우선순위**: P1

#### FR-AUTH-004: 로그아웃
- **설명**: 사용자는 로그아웃하여 세션을 종료할 수 있어야 한다
- **처리**: 클라이언트 측 JWT 토큰 삭제
- **우선순위**: P0

---

### 2.2 선물 이벤트 관리 (Giver 전용)

#### FR-EVENT-001: 이벤트 생성
- **설명**: Giver는 Recipient를 지정하여 새로운 선물 이벤트를 생성할 수 있어야 한다
- **입력**: `name`, `description`, `startDate`, `endDate`, `recipientEmail`
- **출력**: 생성된 이벤트 정보 (id 포함)
- **조건**: 종료일은 시작일 이후여야 함, Recipient는 시스템에 등록된 사용자여야 함
- **우선순위**: P0

#### FR-EVENT-002: 이벤트 목록 조회
- **설명**: Giver는 자신이 생성한 모든 이벤트 목록을 조회할 수 있어야 한다
- **출력**: 이벤트 목록 (name, recipientName, status, createdAt)
- **우선순위**: P0

#### FR-EVENT-003: 이벤트 상세 조회
- **설명**: Giver는 특정 이벤트의 상세 정보, 선물 옵션 목록, Recipient의 선택 내역을 확인할 수 있어야 한다
- **출력**: 이벤트 정보 + GiftOption 목록 + GiftSelection 정보
- **우선순위**: P0

#### FR-EVENT-004: 이벤트 수정
- **설명**: Giver는 자신이 생성한 이벤트의 정보를 수정할 수 있어야 한다
- **입력**: `name`, `description`, `startDate`, `endDate`
- **조건**: COMPLETED 상태의 이벤트는 수정 불가
- **우선순위**: P1

#### FR-EVENT-005: 이벤트 삭제
- **설명**: Giver는 자신이 생성한 이벤트를 삭제할 수 있어야 한다
- **조건**: 관련 GiftOption, GiftSelection 함께 삭제 (Cascade)
- **우선순위**: P1

#### FR-EVENT-006: 수혜자 초대 처리
- **설명**: Giver는 이벤트를 SENT 상태로 변경하여 Recipient에게 선택 권한을 부여할 수 있어야 한다
- **처리**: 이벤트 상태 CREATED → SENT, 초대 링크 생성
- **우선순위**: P0

#### FR-EVENT-007: 구매 완료 처리
- **설명**: Giver는 선물 구매 및 전달 완료 시 이벤트를 COMPLETED 상태로 변경할 수 있어야 한다
- **처리**: 이벤트 상태 SELECTED → COMPLETED
- **우선순위**: P1

---

### 2.3 선물 옵션 관리 (Giver 전용)

#### FR-OPTION-001: 선물 옵션 추가
- **설명**: Giver는 이벤트에 Recipient가 선택할 수 있는 선물 옵션을 추가할 수 있어야 한다
- **입력**: `name`, `description`, `imageUrl`, `points`
- **조건**: 이벤트당 옵션 1개 이상 필수
- **우선순위**: P0

#### FR-OPTION-002: 선물 옵션 삭제
- **설명**: Giver는 등록한 선물 옵션을 삭제할 수 있어야 한다
- **조건**: Recipient가 선택 완료한 옵션은 삭제 불가
- **우선순위**: P1

---

### 2.4 선물 선택 (Recipient)

#### FR-SELECT-001: 초대장 확인 (비로그인 접근 허용)
- **설명**: Recipient는 초대 링크로 접속하여 이벤트 정보와 Giver 정보를 확인할 수 있어야 한다
- **입력**: eventId (URL 파라미터)
- **출력**: 이벤트명, Giver 이름, 초대 메시지, 만료 기간
- **조건**: 인증 없이 접근 가능 (PermitAll)
- **우선순위**: P0

#### FR-SELECT-002: 선물 옵션 열람
- **설명**: Recipient는 이벤트에 등록된 선물 옵션 목록을 확인할 수 있어야 한다
- **출력**: 옵션 목록 (name, description, imageUrl, points)
- **우선순위**: P0

#### FR-SELECT-003: 선물 선택 확정
- **설명**: Recipient는 선물 옵션 중 하나를 선택하여 확정할 수 있어야 한다
- **입력**: `selectedOptionId`
- **처리**: GiftSelection 저장, 이벤트 상태 SENT → SELECTED
- **조건**: 한 이벤트에서 Recipient당 1회만 선택 가능, 이벤트 기간 내에만 선택 가능
- **우선순위**: P0

#### FR-SELECT-004: 선택 결과 조회
- **설명**: Giver 및 Recipient는 선물 선택 결과를 확인할 수 있어야 한다
- **출력**: 선택된 GiftOption 정보, 선택 일시
- **우선순위**: P0

## 3. 비기능적 요구사항 (Non-Functional Requirements)

### 3.1 성능 요구사항

| 항목 | 요구사항 |
|------|----------|
| API 응답 시간 | 일반 조회 API: 평균 500ms 이하 |
| 동시 접속 | 최소 100명 동시 접속 처리 |
| 데이터 조회 | N+1 문제 없는 최적화된 쿼리 사용 (Fetch Join) |

### 3.2 보안 요구사항

| 항목 | 요구사항 |
|------|----------|
| 인증 방식 | JWT (HS256) 토큰 기반 인증 |
| 비밀번호 저장 | BCrypt 해시 암호화 필수 |
| 역할 기반 접근 제어 | GIVER / RECIPIENT 역할별 API 접근 제한 |
| CORS 설정 | 허용된 프론트엔드 Origin만 API 접근 허용 |
| HTTPS | 운영 환경에서 HTTPS 필수 적용 |

### 3.3 가용성 및 신뢰성

| 항목 | 요구사항 |
|------|----------|
| 서비스 가용성 | 99% 이상 uptime |
| 데이터 무결성 | 선물 선택은 중복 처리 방지 (트랜잭션 보장) |
| 오류 처리 | 모든 예외 상황에 일관된 오류 응답 포맷 제공 |

### 3.4 유지보수성

| 항목 | 요구사항 |
|------|----------|
| 코드 구조 | 레이어드 아키텍처 준수 (Controller-Service-Repository) |
| API 문서화 | Swagger/OpenAPI 자동 문서 생성 |
| 도메인 분리 | user, event, option, selection 도메인 단위로 패키지 분리 |

### 3.5 사용성 요구사항

| 항목 | 요구사항 |
|------|----------|
| 반응형 디자인 | 모바일, 태블릿, 데스크톱 모든 환경에서 정상 동작 |
| 초대 링크 | 로그인 없이 초대장 확인 및 이벤트 정보 열람 가능 |
| 에러 메시지 | 사용자가 이해할 수 있는 한국어 에러 메시지 제공 |

## 4. 시스템 인터페이스 요구사항

### 4.1 프론트엔드 ↔ 백엔드 인터페이스
- 통신 프로토콜: HTTP/HTTPS REST API
- 데이터 포맷: JSON
- 인증 헤더: `Authorization: Bearer <JWT>`
- 공통 응답 포맷:
  ```json
  // 성공
  { "success": true, "data": { ... } }
  // 실패
  { "success": false, "error": { "code": "ERR_CODE", "message": "에러 설명" } }
  ```

### 4.2 백엔드 ↔ 데이터베이스 인터페이스
- ORM: Spring Data JPA / Hibernate
- DB: MySQL 8.0+
- 문자셋: utf8mb4 (한국어 및 이모지 지원)
